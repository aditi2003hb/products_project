### ğŸ”¹ 1. **Landing Page (Public View)**

* **What will be shown:**
  * **Two options for the user**:

       * The user fills out a registration form with:
         * **Name**

  * **List of all users** in a table with:

  * **Actions available:**
    * **Add New User**: A form to create new users.
    * **Add Credit/Debit Transaction**: Form to record transactions for existing users.
    * **Enable/Disable Users**: A toggle to change user status (True/False).
    * **Update User Info**: A form to edit user details like location, type, etc.

### ğŸ”¹ 4. **Add Credit/Debit Transaction Page** (`users_credit_debit`)

* **Form Inputs**:

  * The backend checks if the **user exists**.
  * Validates that the **type** matches.
  * Calculates the **balance** and updates the transaction history.
  * The **UUID** is tied to this transaction, and the balance is shown correctly in subsequent logs.

### **2. Backend (Technical Perspective)**

---

#### **SQLite Database Schema**

1. **Users Table:**
   * **Columns**:
     * `uu_id`: `UUID` (Primary Key, auto-generated by backend)
     * `name`: `TEXT`
     * `type`: `TEXT` (From dropdown list: "seller", "buyer", etc.)
     * `location`: `TEXT`
     * `created_date`: `TIMESTAMP` (Auto-generated by backend)
     * `updated_date`: `TIMESTAMP` (Updated automatically when user info is modified)
     * `status`: `BOOLEAN` (True for active, False for inactive)

2. **Transactions Table:**
   * **Columns**:
     * `transaction_id`: `INTEGER` (Primary Key)
     * `user_uu_id`: `UUID` (Foreign Key linked to Users table)
     * `type`: `TEXT` (User type)
     * `credit`: `REAL`
     * `debit`: `REAL`
     * `balance`: `REAL`
     * `date`: `TIMESTAMP`

---

#### **API Endpoints (FastAPI)**

1. **`users_api`** (Add User)
   * **POST request** to create a new user with `name`, `type`, `location`.
   * Backend generates:
     * **UUID**
     * **created_date**

2. **`users_credit_debit`** (Add Credit/Debit Transaction)
   * **POST request** to log credit/debit for an **existing user**.
   * Must match user name and type.
   * Balance is auto-calculated and updated.

3. **`get_users`** (User Management)
   * **GET request** to retrieve all users (accessible only to authorized personnel).
   * Supports:
     * Add user
     * Update user
     * Enable/Disable user (status toggle)
     * List all users

### **4. UUID Full Form:**

* **UUID = Universally Unique Identifier**  
  It's a globally unique identifier used to uniquely identify each user.

---

### **5. Purpose & Why We're Building This**

* **Purpose**:  
  This project is a **user and financial transaction management system**. It allows admins to manage user data, log financial activities (credits and debits), and control user access (enable/disable).

* **Why We're Building It**:  
  This system is needed to handle:
  * User registration and management
  * Financial tracking for users (credit, debit, balance)
  * Admin control over user access (enable/disable)
  * Maintain a **clean backend** and **restricted admin panel** for authorized personnel only.

---

### **6. Project Workflow**

1. **Landing Page (Public)**  
   Shows two options: **Register User** or **Login User**.

2. **Admin Access**  
   Admin enters UUID to access the user management dashboard.

3. **User Registration**  
   Admin or authorized personnel adds a new user via form.

4. **Transactions**  
   Admin logs credit/debit transactions for existing users.

5. **User Management**  
   Admin can update user info, enable/disable users, and view all users.
